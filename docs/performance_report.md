## 1. 調査目的

本レポートは、SDND（Spec-Driven Novel Development）アーキテクチャの **パフォーマンス上の制限と限界** を調査し、以下の問いに答えることを目的とする。

- このまま執筆を続けた場合、何話まで執筆可能か？
- 各種設定ファイルはどのくらいまで増やせるか？
- ボトルネックはどこにあるか？

## 3. コンテキストウィンドウ負荷分析

### 3.1 TIER別ローディングコスト

SDNDアーキテクチャでは、エピソード執筆時に以下のファイル群をコンテキストに読み込む。

#### TIER 1（常時読み込み・必須）

| ファイル | サイズ | 用途 |
|---------|--------|------|
| specs/core/invariants.md | 5,772 bytes | 不変ルール（INV-*） |
| specs/core/characters_current.md | 12,892 bytes | キャラクター現在状態 |
| canon/quick_ref.md | 3,923 bytes | Canon最新スナップショット |
| **合計** | **22,587 bytes（約22 KB）** | |

#### TIER 2（エピソード実行時に読み込み）

| ファイル | サイズ | 用途 |
|---------|--------|------|
| canon/active/*.md（5ファイル） | 82,123 bytes | 直近5話のCanonエントリ |
| meta/open_loops.md | 40,148 bytes | オープンループ一覧 |
| **合計** | **122,271 bytes（約119 KB）** | |

#### TIER 3（必要時のみ参照）

| ファイル | サイズ（概算） | 用途 |
|---------|-------------|------|
| specs/reference/world.md | 約5 KB | 世界設定 |
| specs/reference/magic_physics.md | 約9 KB | 魔法体系 |
| specs/reference/characters_full.md | 約14 KB | キャラクター詳細 |
| docs/plot_outline.md | 約15 KB | プロット概要 |
| 前話のEP本文 | 約16〜33 KB | 文体・引き継ぎ参照 |
| **合計（概算）** | **約60 KB** | |

#### 1エピソード執筆時の総ローディング

| 層 | サイズ | 読み込み頻度 |
|----|--------|------------|
| TIER 1 | 約22 KB | 毎回必須 |
| TIER 2 | 約119 KB | 毎回必須 |
| TIER 3 | 約60 KB | ほぼ毎回 |
| エージェント定義 | 約10 KB | 毎回 |
| **合計** | **約211 KB** | |

Claude Opus 4.6のコンテキストウィンドウ（200Kトークン）に対して、約211 KBのテキストは **約50K〜70Kトークン** に相当する（日本語テキストは1トークンあたり約3〜4バイト）。これはコンテキストの **25〜35%** を占める。残り65〜75%が実際の執筆・推論に使える。

### 3.2 ファイルサイズの推移傾向

#### エピソード本文

| 範囲 | 平均サイズ | 最小 | 最大 |

#### characters_current.md（TIER 1で最大のファイル）

| 時点 | サイズ |

## 4. スケーラビリティ分析

### 4.1 ボトルネック候補

| ファイル | 現在サイズ | 成長率（/話） | 100話時点推定 | 危険度 |
|---------|-----------|-------------|-------------|--------|
| open_loops.md | 40 KB | 約1.1 KB/話 | 約112 KB | **高** |
| characters_current.md | 13 KB | 約285 B/話 | 約31 KB | 中 |
| canon/active/（合計5件） | 82 KB | 横ばい（窓制御） | 約90 KB | 低 |
| quick_ref.md | 4 KB | 微増 | 約6 KB | 低 |
| invariants.md | 6 KB | ほぼ横ばい | 約7 KB | 低 |

### 4.2 推定キャパシティ

#### シナリオ A: 現行アーキテクチャのまま

| 話数 | TIER 1+2 合計 | コンテキスト占有率 | 評価 |
|------|-------------|-----------------|------|
| 35話（現在） | 約145 KB | 約25〜35% | 快適 |
| 50話 | 約165 KB | 約30〜40% | 快適 |
| 75話 | 約195 KB | 約35〜45% | やや圧迫 |
| 100話 | 約230 KB | 約40〜55% | 圧迫域 |
| 150話 | 約290 KB | 約55〜70% | 限界域 |
| 200話 | 約350 KB | 約65〜85% | 執筆困難 |

**結論: 現行アーキテクチャのまま、約100話までは問題なく執筆可能。150話前後で限界に近づく。**

#### シナリオ B: ボトルネック対策を実施した場合

以下の対策を行えば、さらに延長できる。

1. **open_loops.md のアーカイブ分離**（効果: 最大）
   - resolvedループの詳細履歴を別ファイルに移動
   - activeループ + 直近resolvedのサマリーのみ保持
   - 推定削減: 30〜35 KB → open_loops.md を 5〜10 KB に維持可能

2. **characters_current.md の圧縮**（効果: 中）
   - 非アクティブキャラの詳細をTIER 3に移動
   - 「直近で登場したキャラ」のみTIER 1に保持
   - 推定削減: 5〜8 KB

3. **Canon active window の削減**（効果: 中）
   - 5ファイル → 3ファイルに縮小
   - 推定削減: 約30 KB

対策実施後の推定:

| 話数 | TIER 1+2 合計 | コンテキスト占有率 | 評価 |
|------|-------------|-----------------|------|
| 100話 | 約160 KB | 約30〜40% | 快適 |
| 200話 | 約200 KB | 約35〜50% | やや圧迫 |
| 300話 | 約250 KB | 約45〜60% | 圧迫域 |

**対策実施後は約200話まで快適に執筆可能。300話前後が理論上の限界。**

## 5. その他のパフォーマンス観測

### 5.1 エピソードパイプラインの処理特性

1話あたりのパイプライン（branch → write → QA → Canon → merge）で、以下の作業がコンテキストを消費する:

| 工程 | 主な消費 | 備考 |
|------|---------|------|
| writer | TIER 1+2+3読み込み + 本文生成 | 最大消費。1回のエージェント呼び出し |
| qa | TIER 1+2読み込み + EP本文 + レポート生成 | writer に次ぐ消費 |
| editor | Canon proposal + specs読み込み + 更新 | 比較的軽量 |

各工程はサブエージェントとして独立したコンテキストで実行されるため、**PMセッション自体のコンテキスト枯渇は発生しにくい**。ただしPMセッション内で複数話を連続処理すると、セッション全体の会話履歴が蓄積し、PMコンテキストの圧縮（自動要約）が発生する。

### 5.2 Git リポジトリの成長

| 項目 | 35話時点 | 100話推定 | 200話推定 |
|------|---------|----------|----------|
| コミット数 | 139 | 約400 | 約800 |
| .git サイズ | 5.1 MB | 約15 MB | 約30 MB |
| 総ファイルサイズ | 8.2 MB | 約20 MB | 約40 MB |

Gitリポジトリのサイズは実用上の問題にならない。

### 5.3 QAレポートの蓄積

QAレポートは1話あたり約25 KBで蓄積される。100話で約2.5 MB。これらはTIER読み込みの対象外のため、コンテキスト負荷に影響しない。ただしディスク容量の観点では無視できる規模。

## 6. 設計上の強みと限界

### 6.1 強み

| 設計要素 | 効果 |
|---------|------|
| Canon 3層分離（quick_ref / active / archive） | activeウィンドウが5ファイルに制限され、読み込み量の天井がある |
| TIER階層化 | 必要なファイルのみ読み込み、不要な参照コストを排除 |
| サブエージェント分離 | 各工程が独立コンテキストで実行され、PMセッションの枯渇を防ぐ |
| canon-proposal パターン | writerが生成 → editorが登録する分業で、仕様の一貫性を保つ |

### 6.2 限界

| 要素 | 制限の内容 | 影響が出始める時期 |
|------|-----------|-----------------|
| open_loops.md の線形成長 | resolved履歴が無制限に蓄積 | 約75〜100話 |
| characters_current.md の成長 | キャラ数・状態記述の増加 | 約100〜150話 |
| TIER 1の固定読み込み | 毎回全量読み込みのため圧縮が難しい | 150話以降 |
| Canonエントリのサイズ増大 | 1エントリ16〜17 KB（Arc 3後期） | 間接的影響 |

## 7. 推奨事項

### 即時対応不要（現在の35話時点）
現在のアーキテクチャは35話の規模に対して十分な余裕がある。直近の改修は不要。

### 50話到達時に検討
- open_loops.md のresolved履歴をアーカイブ分離する仕組みの設計

### 100話到達時に実施
- open_loops.md アーカイブ分離の実装
- characters_current.md の「アクティブキャラのみ」フィルタリング
- Canon active window サイズの再評価

### 長期的な改善案
- TIER 1ファイルの自動圧縮メカニズム（要約生成）
- エピソード間の差分のみを伝達する仕組み（full reload回避）
- open_loops.md を構造化データ（JSON/YAML）に移行し、クエリベースで必要部分のみ抽出

---

## 8. 結論

SDNDアーキテクチャは **100話程度まで現行のまま安定して運用可能** であり、ボトルネック対策を施せば **200話以上** にも拡張できる。最大のボトルネックは `open_loops.md` の線形成長であり、resolved履歴のアーカイブ分離が最も効果的な対策となる。

35話完結時点でのコンテキスト占有率は25〜35%と十分に余裕があり、本プロジェクトのアーキテクチャ設計は長編小説の執筆に適していることが実証された。
