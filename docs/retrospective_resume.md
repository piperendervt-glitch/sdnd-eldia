# 前プロジェクト（magic-debug-isekai-spec）振り返りサマリー

## プロジェクト概要
- 仕様駆動（SDND）の異世界転生ライトノベル執筆プロトタイプ
- EP-001 〜 EP-010 の10話を完了
- 主人公：現代のソフトウェアエンジニアが異世界に転生し、デバッグスキルで魔法体系のバグを発見・修正する

## 達成したこと
- 10話分のエピソード執筆ワークフローを確立
- canon.md による正史管理の基本パターンを構築
- QA レポートによる品質チェックプロセスを運用
- 不変ルール（invariants）による世界設定の一貫性維持

## スケーリング分析で判明した問題

### 問題1: canon.md の線形成長
- 成長率: ~2,050文字/EP
- EP-68 で 200K トークン上限に到達する見込み
- **対策**: canon を3層分離（quick_ref / active / archive）

### 問題2: 全スペック一括読み込み
- EP-10 時点で全スペック ~37K tokens
- エピソード執筆に不要な設定も毎回ロード
- **対策**: specs を TIER 1（必須）/ TIER 2（実行時）/ TIER 3（参照時）に階層化

### 問題3: open_loops の放置
- EP-008 以降 open_loops が未更新
- canon との同期が崩壊
- **対策**: Canon 承認のブロッキングゲートとして同期チェックを制度化

### 問題4: perf_log の空欄
- elapsed_seconds が全件空欄（手動入力に依存していた）
- **対策**: bash スクリプトによる自動記録、手動入力禁止

## 継承するパターン
- エピソード → QA → Canon 確定のワークフロー
- 不変ルール（INV-*）による制約管理
- キャラクターID体系（PC-*, NPC-*）
- ループID体系（LOOP-*）
- CANON エントリID体系（CANON-*）

## 継承しないもの（改善対象）
- 単一 canon.md ファイル → 3層分離へ
- スペック一括読み込み → 階層化へ
- open_loops の手動同期 → ゲート制度化へ
- perf_log の手動記録 → 自動記録へ
