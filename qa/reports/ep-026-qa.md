### [Major] M-001: MP計算の時系列不整合（canon-proposal の終了MP値が誤っている可能性）

- 該当箇所: story/arc3/ep-026.md L7（ヘッダ）、story/arc3/ep-026.canon-proposal.md「MP計算の根拠」欄
- 参照仕様: INV-A01 / RULE-M01 / canon/active/canon-025.md（EP-025終了状態）

**問題の詳細:**

本文の時系列を精査すると、EP-026 は以下の日数にわたる出来事を描いている。

| 日 | 本文の根拠 | 行番号 |
|----|-----------|--------|
| Day 0 (EP-025 最終日) | クラウスへ面会申請。「翌日返事待ち」 | canon-025 EVT-025-06 |
| Day 1 | 「クラウスからの返事は昼前に来た」「明日の朝、研究室に来なさい」 | L19, L21 |
| Day 2 | 「翌朝」クラウス研究室で面会。クラウスがベルントへ申請 | L23 |
| Day 3 | 「ベルントからの返答は翌日の夕方に来た」 | L141 |
| Day 4 | 「翌朝の七時半に」封印区画へ入室・発動 | L145 |

MP回復の正確な計算:
- EP-025 終了時: 77/120（canon-025 確定値）
- Day 0〜3 で 3日分の自然回復（入室当日の朝は Day 4 だが、回復は前日就寝までと解釈）: 7MP/日 × 3日 = 21MP
- EP-026 入室前: 77 + 21 = **98/120**
- structural_overview 発動消費: 22MP
- EP-026 終了時: 98 - 22 = **76/120**

canon-proposal の計算は「1日の自然回復 +7MP → 84/120 開始 → 62/120 終了」としており、実際の時系列（Day 4 に入室）と乖離している。

ヘッダ `<!-- 季節: 初夏（EP-025から翌日）。アル12歳 -->` の「EP-025から翌日」という記述が、MP計算の根拠を「1日回復」とした原因と考えられるが、実際の本文では入室は Day 4 で、canon 確定値として登録されると以降の計算が14MP分ずれる。

**修正提案:**
1. 本文ヘッダ（L7）を `初夏（EP-025から数日後）` に修正する
2. canon-proposal の「MP計算の根拠」を以下に修正する:
   - EP-025 終了時: 77/120
   - 3日間の自然回復: 7MP/日 × 3日 = 21MP
   - EP-026 入室前: **98/120**
   - structural_overview 発動消費: 22MP
   - EP-026 終了時: **76/120**

### [Minor] M-002: デバッグ出力の layer topology における表記の曖昧さ

- 該当箇所: story/arc3/ep-026.md L194
- 参照仕様: INV-D01 / canon/active/canon-023.md「新規モジュール名・構造の確定情報」

**問題の詳細:**

EP-026 のデバッグ出力に以下の行がある。

```
    [L2] mana_routing (sub: world_management_system)
         — post_collapse_rebuild version
```

確定した canon（canon-023・canon-025）では `mana_routing` は `world_management_system` の *サブモジュール* として確定している（`mana_routing → sub_module of world_management_system`）。

しかし `mana_routing (sub: world_management_system)` という括弧表記は「mana_routing のサブに world_management_system がある」という読み方も成立し、上位・下位の階層関係が逆に読める可能性がある。

本文 L79 での口頭説明（「`entity_registry`と`mana_routing`はその（world_management_system の）配下のサブモジュールとして出力されました」）は正確だが、デバッグ出力テキスト自体の表記と齟齬が生じている。

**修正提案:**

以下のいずれかに修正することで、canon と整合した明確な表記となる。

案1（上位モジュールを `sub:` の代わりに `in:` で示す）:
```
    [L2] mana_routing (in: world_management_system)
         — post_collapse_rebuild version
```

案2（階層を明示）:
```
    [L2] mana_routing [sub_module of: world_management_system]
         — post_collapse_rebuild version
```

### [Warn] W-001: 発動時に「目を閉じた」描写と INV-D01 の視認条件の関係

- 該当箇所: story/arc3/ep-026.md L161, L171
- 参照仕様: INV-D01（「発動条件: 対象の魔法現象を視認している状態であること」）

**問題の詳細:**

L161: `アルは部屋に入ってすぐ、中央に立って壁を確認した。`（魔法陣の視認が完了）
L171: `アルは目を閉じた。`（発動直前）

INV-D01 の発動条件は「対象の魔法現象を視認している状態であること」だが、目を閉じた後に発動している。

仕様には「認識段階で視認する必要があるが、発動中は視認を維持しなくてよい」という明示的な定義がなく、RULE-M08「認識 → 構築 → 発動」の流れでも目を閉じることの可否は明示されていない。

「前世の仕事で言えば、ブロック図を眺める感覚」（L173）という内省から、「引いた視点で見る」ためにあえて目を閉じたと解釈できる。また、過去エピソード（EP-023・EP-024）では目を閉じる描写がなかったため、この描写が初出の表現パターンとなっている。

**修正提案:**

修正は必須ではない（Warn レベル）。ただし以下のいずれかを検討することを推奨する。

案1（仕様上の明示で対応）: RULE-M08 に「認識段階で視認を確立すれば、構築・発動フェーズでの目の開閉は制約しない」という補記を追加する。

案2（本文の修正で対応）: `アルは目を閉じた。` を `アルは一度壁を見てから、視線を内側に向けた。` 等に変更し、視認を維持したまま「内向き」の感覚に切り替えたことを明示する。

### [Warn] W-002: canon-proposal の「mana_routing の層位置（L2）」について

- 該当箇所: story/arc3/ep-026.md L194, story/arc3/ep-026.canon-proposal.md「世界状態の変化」
- 参照仕様: canon/active/canon-023.md「新規モジュール名・構造の確定情報」

**問題の詳細:**

EP-026 の出力 layer topology では以下の層位置が設定されている。

```
  layer topology (partial):
    [L0] foundation_layer — base substrate
    [L1] routing_table — surface management layer
    [L2] mana_routing (sub: world_management_system)
```

canon-023 では `routing_table` が `surface-level management layer over deeper foundation schema` として確定しており、EP-026 の `[L1] routing_table — surface management layer` と一致する。

しかし `mana_routing` が `[L2]` に配置されている点について、canon-023 では `mana_routing` は `foundation_layer` 内の `world_management_system` のサブモジュールとして記録されている。これは `[L0] foundation_layer` の内部モジュールであるため `L2` 配置（foundation_layer や routing_table より「上」の層に見える）は直観的に不自然である。

「layer topology (partial)」という注記があるため全体構造を完全に示していない表現であることは明示されているが、この topology 表記が canonically な層構造と矛盾しているように読める点は将来の参照時に混乱を招く可能性がある。

**修正提案:**

canon-026 登録時に、layer topology の読み方について備考を明記することを推奨する。例:「`[L2]` の mana_routing は `[L0] foundation_layer` の内部モジュールであり、routing_table（L1）の上位層ではなく、別系統の深層モジュールとして位置付けられる」等の補足。本文修正は必須ではない。
