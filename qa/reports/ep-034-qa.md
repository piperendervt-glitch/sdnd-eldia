### [Minor] QA-034-M01: 28MP消費内訳とRULE-M08基礎コスト（15MP）の関係が不明確

- 該当箇所: story/arc3/ep-034.md L422（エピソードメモ）、story/arc3/ep-034.canon-proposal.md L47-55
- 参照仕様: RULE-M08（デバッグスキルの技術仕様・MP消費）
- 詳細:
  RULE-M08によると、surface_bleed直接接触によるスキル発動には「基礎コスト15MP + 対象の複雑さに応じた追加コスト（5〜50MP）」が適用される。また「バグ修正コストは解析コストとは別に計上する」と規定されている。

  EP-034での消費内訳はエピソードメモL422で「node #7: 10MP、node #3: 6MP、node #11: 6MP、node #14: 6MP、計28MP」と記述されているが、RULE-M08の基礎コスト（15MP）がこの内訳にどう対応するかの説明がない。

  EP-033では「基礎15MP + 追加10MP = 25MP」という明確な内訳だった（canon-033.md L105-106）のに対し、EP-034では4ノードへのwrite操作で計28MPという数値は、RULE-M08の「1接続あたり基礎15MP」の枠組みで説明するには過小の可能性がある。

  ただし「root_access権限下では接続コスト構造が変化する」という記述も仕様にはなく、物語的には本文の消耗描写（膝をついた描写、意識保持ぎりぎりの感触）から消耗の重大さは十分伝わっている。
- 修正提案:
  選択肢1: エピソードメモのMP内訳を「接続維持コスト（RULE-M08基礎コスト相当）+ 各ノードwrite操作コスト」の二層構造で明示する（例: 接続維持3MP + node #7: 10MP + node #3/#11/#14各5MP = 合計28MP、計算上は整合）。
  選択肢2: 現状維持。「root_access権限下での多点write操作は通常の単一解析と異なるコスト構造を持つ」として本文の疲労描写で補う。物語的な問題は小さい。
  推奨: 選択肢2（現状維持）。本文の消耗描写が十分に機能しており（膝をついた・意識保持ぎりぎり・足の重さ・喉の渇き・体温上昇）、物語上の支障はない。canon-033の備考にも「write操作を伴う場合は追加コストが増大する可能性」の参照注記あり。

### [Warn] QA-034-W01: INV-D01「一対象制約」の拡張解釈について確認が望ましい

- 該当箇所: story/arc3/ep-034.md L74-218（node #7接触からnode #3/#11/#14への順次write操作）
- 参照仕様: INV-D01（デバッグスキルの基本制約）、RULE-M08「解析と修正における一対象の定義」
- 詳細:
  INV-D01は「一度に解析できる対象は一つのみ」と規定している。EP-034では node #7 への直接接触（surface_bleed）で開いた1つのsupervisor_channelセッション内で、#3, #11, #14 へのwrite操作も実行している。

  これについてcanon-proposal（L138）では「一つの露出点から一つのsupervisor_channelセッション内で実行として整合」と説明しており、EP-033設計（canon-033 EVT-033-08）でも同様の構造が既に承認されている。

  RULE-M08の「一対象の定義」補則では「修正フェーズでは修正を加える発振源ひとつを一対象として扱う」と規定されているが、本話の描写ではnode #7 → #3 → #11 → #14と順次個別に操作しており、この逐次処理形式はRULE-M08の二段階処理（全体解析→単体修正）の拡張として整合的に読める。

  実際の本文でも各ノードの測定・フィルタ適用が個別に描かれており（L153「node #3の断裂境界での位相オフセットを測定した上で適用する」等）、「一度に4ノードを同時処理」ではなく「逐次・個別処理」として描写されている点は評価できる。

  Blockerではないが、Canon登録時に「root_access権限下でのsupervisor_channel経由の多点逐次操作はINV-D01に抵触しない」という解釈をRULE-M08またはspecsに明記することを検討されたい。
- 修正提案:
  Canon登録後の任意作業として、RULE-M08またはspecs/reference/magic_physics.mdに「root_access権限付与後のsupervisor_channelセッション内での逐次write操作はINV-D01の一対象制約と整合する（一操作ごとに一対象）」という補足を追記することを推奨する。
