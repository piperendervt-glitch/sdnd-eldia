## 魔法体系ルール一覧（RULE-M01〜M08）

### RULE-M01: マナ（MP）の基本原理
- すべての生物はマナを体内に保有する（量には個人差がある）
- マナは魔法行使の燃料であり、消費すると疲労・眩暈などの身体症状が現れる
- マナの自然回復は睡眠・休息によって行われる（完全回復には一晩の睡眠が必要）
- マナ残量がゼロになると「マナ枯渇」状態となり、意識喪失・最悪の場合は生命の危機
- INV-A01（魔法にはコストがある）に準拠
- **補記（EP-009 QA W01対応）**: 「一晩の睡眠で完全回復」は健常時または通常消費（総MPの70%未満）に対して適用される。**極度のマナ消耗（総MPの70%以上を一度に消費した場合）からの回復は、回復速度が低下し数日〜1週間を要する**。EP-008での実績（105MP消費、総MP120の87.5%に相当）がこの規定の適用事例。5日間の安静で70/120（約58%）まで回復したことを確定事実として記録（CANON-009参照）。

### RULE-M02: 魔法の発動プロセス
- 魔法の発動は「認識 → 構築 → 発動」の3段階で行われる
  1. **認識**: 術者が魔法の効果をイメージし、必要なマナ量を感知する
  2. **構築**: マナを体内で魔法的パターンに変換する（詠唱・魔法陣はこの補助手段）
  3. **発動**: 構築されたパターンを外部に放出し、効果を実現する
- 各段階で中断可能（ただし構築段階以降の中断はマナの一部を消費する）
- 熟練者は認識と構築を高速化できる（無詠唱の条件）

### RULE-M03: 属性体系
- 魔法は6つの基本属性に分類される

| 属性 | 対応する力 | 相克関係 |
|------|-----------|----------|
| 火 | 熱・燃焼 | 水に弱く、風で強化 |
| 水 | 流動・冷却 | 火に強く、土に弱い |
| 土 | 固化・重力 | 水に強く、風に弱い |
| 風 | 運動・切断 | 土に強く、火を強化 |
| 光 | 照明・回復・浄化 | 闇と相互打消し |
| 闇 | 隠蔽・腐食・強化 | 光と相互打消し |

- 無属性魔法も存在する（身体強化など）
- 属性の相克は絶対的な優劣ではなく、同程度の魔力なら有利不利が生じる程度
- 複合属性（例: 火＋風＝爆炎）は上級者のみ可能
- INV-A03（属性間の相互作用は固定法則に従う）に準拠

### RULE-M04: 魔法の射程と範囲
- 魔法の射程は術者のマナ量と熟練度に依存する
- 射程が長いほど、範囲が広いほど、MP消費は増大する
- 術者の視認範囲を大きく超える射程の魔法は精度が著しく低下する
- 範囲魔法は味方を巻き込むリスクがある（制御には高い技量が必要）

### RULE-M05: 魔法の干渉と相殺
- 同属性の魔法同士がぶつかると、マナ量の多い方が残る（差分のみ）
- 相克属性の魔法同士は相互に威力を減じ合う
- 異なる術者の魔法を意図的に合成するには、両者の高度な連携が必要
- 同一空間に複数の魔法が密集すると「魔法干渉」が発生し、予期しない効果が生じる可能性がある

### RULE-M06: 詠唱と魔法陣
- **詠唱**: 魔法構築（RULE-M02 の第2段階）を言語で補助する技法。安定性が増す
- **魔法陣**: 地面や空中にマナで描く幾何学パターン。複雑な魔法や儀式魔法に必要
- 詠唱を省略する「無詠唱」は、魔法の理論を完全に理解した術者のみ可能
- 魔法陣は事前に描いておくことで発動を高速化できる（罠や防衛陣として利用）
- INV-A04（魔法の効果は術者の理解度に依存する）に準拠

### RULE-M07: 魔法体系の「バグ」
- 世界の魔法体系には「仕様通りに動作しない」不整合が存在する
- バグの種類:
  - **例外バグ**: 特定条件下で魔法が想定外の挙動をする
  - **境界バグ**: 属性の相克表に当てはまらない境界ケース
  - **オーバーフロー**: マナの過剰集中による暴走現象
  - **ヌル参照**: 魔法の対象が存在しない場合の未定義動作
- バグは自然界にも影響を与えており、「魔法的に不安定な場所」として認知されている
- 一般の魔法使いはバグを「魔法の気まぐれ」や「禁域」として避けている
- 主人公のデバッグスキルのみがバグの構造を正確に可視化できる

### RULE-M08: デバッグスキルの技術仕様
- **発動条件**: 対象の魔法現象を視認している状態（RULE-M02 の認識段階に相当）
- **MP消費**: 基礎コスト 15MP + 対象の複雑さに応じた追加コスト（5〜50MP）
- **出力形式**: 術者（主人公）の視界にプログラミング的メタファーで魔法構造が表示される
  - 変数名 → マナの流れのラベル
  - スタックトレース → 魔法の因果連鎖
  - エラーメッセージ → バグの症状説明
  - ソースコード → 魔法の構造式
- **解析時間**: 単純な魔法で数秒、複雑な体系で数分〜数十分
- **制約**:
  - 一度に解析できる対象は一つのみ（INV-D01）
  - 「見る」だけで「直す」ことはできない（修正には理論理解が必要）
  - 解析中は他の魔法を行使できない（集中が必要）
  - バグの修正は不可逆（INV-D02）
- **不随意発現時のコスト**:
  - 不随意発現（対象の魔法を視認した際の無意識的な反応）時は基礎コスト（15MP）のみ適用。追加コストは意識的発動時にのみ発生する
  - 不随意発現時は術者による精度制御が働かないため、対象の複雑さに依らずコストが固定される
  - 意識的発動が確立した後（EP-004 以降）は通常のコスト計算（基礎コスト + 追加コスト）が適用される
- **大規模魔法異常における受動的感知漏れ**（EP-007 QA m01-R 対応・EP-008 QA W01 対応）:
  - forest-level 以上（広域に影響を及ぼす大規模魔法異常）の場合、術者が異常の方向を視覚的に向いている状態（視線方向に魔法現象が存在する状態）であれば、能動的な発動意図なしに感覚に干渉してくる「受動的感知漏れ」が発生することがある
  - これは EP-003 の不随意発現の応用であり、異常規模が大きすぎて方角を向いているだけで感覚に干渉した形式として扱う
  - 受動的感知漏れ時のコストは不随意発現時と同様に基礎コスト（15MP）のみ適用
  - 受動的感知漏れはINV-D01の能動的スキル発動（視認必須・一対象・MP消費あり）とは区別する。スキルを「自動反応している状態」と「能動的に発動した状態」の二段階として処理することが可能
- **バグ修正コスト**（EP-008 QA m01 対応）:
  - バグ修正操作のMP消費は解析コストとは別に計上する
  - 修正対象の複雑さ・修正操作の規模に応じて 15〜80MP 程度が目安
  - 初修正・単一発振源への操作: 50〜70MP が目安（EP-008実績: 60MP）
  - 修正コストは INV-A01「魔法の規模・複雑さに比例するMP消費」の一般規定に準拠する
- **解析と修正における「一対象」の定義**（EP-008 QA W04 対応）:
  - 「一つの対象」とは術者が意識的に焦点を当てている魔法現象の一単位を指す
  - 複数発振源が一体として作用している複合体（例: 七重フィードバックループ）は、全体の構造把握を目的とする解析フェーズでは一対象として扱うことができる
  - 修正フェーズでは修正を加える発振源ひとつを一対象として扱う（解析で全体構造を把握した上で、修正は単一要素のみに作用する）
  - この二段階処理（全体解析 → 単体修正）はINV-D01の「一対象制約」と整合する
- **surface_bleed直接接触によるアクセス**（EP-030 初確認）:
  - 地上に露出したBOUNDARY_LEAKの露出点（surface_bleed）に直接触れることで、封印区画への入室と同等のfoundation_layerへのアクセスが可能
  - 条件: prior_registered debugger_accessのアクセストークン（prior_access_pattern）が必要
  - BOUNDARY_LEAK拡大に伴いこのルートが新たに成立した（スキルレベルの向上ではなく世界側の変化による。INV-C01準拠）
  - INV-D01の「対象の魔法現象を視認している状態」の接触による等価的適用として処理する
- **supervisor_channel経由でのクエリ送信**（EP-030 初確認・QA W-03対応）:
  - prior_registered debugger_accessのアクセストークンを持つアクセスに対して`world_management_system`の`supervisor_channel`が開放された場合、デバッグスキルの発動維持中に内部意図（思考内容）をクエリとして対象システムに送信することが可能
  - これはlux_management_coreが準備したチャンネルへの双方向通信であり、デバッグスキル単独の能力拡張ではない（INV-C01の成長の段階性に違反しない）
  - supervisor_channelはlux_management_coreとの直接テキスト対話を可能にする専用チャンネル

## リンデン地域バグの定義（EP-012 事前設計）

### 根本原因

EP-008 のバグ修正（逆位相注入によるアクティブ・ノイズキャンセリング）の副作用（INV-D02）で、リンデン地域のマナ経路テーブルが破損した。マナの属性別配送ルートを管理するシステム層（SF真相: ルーティングテーブル）が部分的に壊れ、各地で異なる症状として発現している。

### 症状マップ

| 場所 | 症状 | メカニズム |
|------|------|----------|
| 魔法塾付近 | ハーブ枯れ（EP-009 ハンナ証言、EP-010 直接確認） | マナ供給経路が細り、環境マナに依存する植物が餓死 |
| 魔法塾内 | 火・土属性の出力低下（EP-010 フィン報告） | 該当属性チャネルが劣化し、術者の魔法発動に対するレスポンスが低下 |
| クロッカ丘陵南斜面 | 属性境界バグ: 土・水の遷移帯崩壊（EP-011 確定） | 境界フィルタが機能不全。両属性のマナを植物が過剰吸収し結晶化 |

### EP-012 修正設計

- **修正対象**: 魔法塾近くのマナ経路テーブル破損（配送ルートの部分再構成）
- **MP消費見込み**: 解析 20MP + 修正 50MP = 合計 70MP（MP 80/120 開始 → 10/120 で極度消耗）
- **副作用（INV-D02）**: 修正でマナの流れが回復した結果、町の南側・農地方面でマナの過剰流入が発生。農作物の異常成長や夜間に地面が淡く光る現象。目に見える規模の副作用を町の人々が目撃する → EP-014 の召喚への因果
- **修正の限界**: アルの権限（authority）では完全修復は不可能。部分修復のみ

### デバッグ出力フォーマット（EP-012 用）

```
[SCAN: regional_mana_routing]
routing_table: CORRUPTED (entries: 47, invalid: 12)
cascade_origin: external_patch (ref: forest_resonance_fix)
attribute_channels: fire[degraded], earth[degraded], water[overflow], wind[nominal]
note: partial_repair possible — full restoration requires elevated authority
```

EP-004（NullRef・overflow）、EP-008（七重フィードバックループ）、EP-011（boundary_violation: attribute_transition_error）とは異なる種類のバグとして、routing_table タグを使用。「elevated authority」は EP-011 の「insufficient authority」からの連続性を持ち、LOOP-001・LOOP-007 への伏線としても機能する。

## 魔法の分類（用途別）

| 分類 | 説明 | 例 |
|------|------|-----|
| 攻撃魔法 | 対象にダメージを与える | ファイアボール、ウィンドカッター |
| 防御魔法 | 障壁を展開する、攻撃を逸らす | マナシールド、アースウォール |
| 回復魔法 | 傷を癒す、状態異常を解除する | ヒール、キュア |
| 強化魔法 | 身体能力・魔法耐性を一時的に向上 | ブースト、マナアーマー |
| 探知魔法 | 周囲の情報を取得する | マナセンス、トラッキング |
| 生活魔法 | 日常生活に用いる低コスト魔法 | ライト、クリーン、イグナイト |

## H7: 祈りの古語のコマンド構文設計（EP-016 事前設計）

### 概要

ルクス教の祈りの言葉は、時代を経て「自然言語化」したシステムコール（管理AI「ルクス」への命令）の名残である。大聖堂の古聖典には、コマンド構文に近い原形が残っている。

### 現在の祈り（標準祈祷書）

「ルクスよ、光をお与えください。闇を退け、道を照らしたまえ」
— 自然言語的な祈り。儀式・礼拝で使用される標準的な祈祷文

### 古い祈り（古聖典・制限区画の文書）

「ルクシア・エーテル・コンディーレ、シ・テネブラ・エスト、トゥンク・ルーメン・ファキアート」
（Luxia ether condere, si tenebra est, tunc lumen faciat）

### アルの解析（EP-016 での着地）

アルが古聖典の音韻構造を前世知識で分析した場合の内心描写:

- `luxia_ether_condere` → ルクスの基盤を確立する（INIT / 初期化処理）
- `si_tenebra_est` → もし闇があるならば（IF condition / 条件分岐）
- `tunc_lumen_faciat` → そのとき光を生成せよ（THEN action / 実行命令）

アルの内心:（これは……if-then文だ。条件分岐だ）→ 即座に「考えすぎだ。古い言語の語順が偶然似ているだけだ」と打ち消す（INV-B03 準拠。SF真相は未開示）

### SF真相との整合

- 古い祈りは管理AI「ルクス」へのシステムコールの原形
- 時代を経て住民が意味を理解しないまま口承で伝え、「自然言語化」して現在の祈祷書の形になった
- 古聖典にはコマンド構文に近い原形が残っている
- H6（EP-003: 回復魔法の波形にプロトコルヘッダ）との連続性: H6は「魔法発動時の信号」にプロトコルが含まれる発見、H7は「祈りの言語構造自体」がコマンドである発見。いずれも「魔法＝システムのAPIコール」という真相の断片

### 描写の指針

- アルがH7に気づくのはEP-016。大聖堂図書室の古聖典を閲覧した際
- 発見のきっかけ: 学院長クラウスが「魔法の歴史を学ぶなら大聖堂の図書室が良い」と案内し、ベルントも「教養として知っておくべき」と許可。両者の思惑は異なるが結果的にアルのアクセスが実現
- 古聖典の制限区画へのアクセスには司教以上の紹介状が必要 → ベルントまたはクラウスの口添え
- アルの反応は「気づく→打ち消す」のパターン（H4と同様）。ただしH4（世界地図）よりも具体的なパターン認識であるため、打ち消しにやや苦労する描写が望ましい
